# TES Strategy Workflow

This document describes how strategy-driven simulations run in TES today, which strategy command is currently available, and what strategy workflow improvements are planned next.

## Strategy interface

TES strategies implement the `Strategy` interface and define two lifecycle hooks:

- `on_start() -> list[TesCommand]`
- `on_event(event: TesEvent) -> list[TesCommand]`

Both methods return strict command models (`TesCommand` variants), not raw dictionaries. This keeps strategy output aligned with TES command contracts.

### `on_start()`

`on_start()` runs once at the beginning of a strategy-driven simulation and returns the initial command queue to submit to the engine.

Use `on_start()` for deterministic startup intent such as:

- posting initial orders,
- seeding baseline inventory behavior,
- establishing the first step in a repeatable scenario.

### `on_event(event)`

`on_event(event)` runs for each emitted event generated by the engine after command execution. It receives a strict `TesEvent` model and can return additional strict commands to extend the run.

Use `on_event(event)` for deterministic reactive behavior such as:

- replacing filled orders,
- canceling stale working orders,
- adjusting next commands based on observed event flow.

## Strict command/event model flow

TES strategy runs follow a strict model pipeline:

1. CLI loads a named strategy implementation.
2. Strategy `on_start()` returns initial `TesCommand` values.
3. Simulation executes each command through the engine adapter.
4. Engine emits strict event objects consumed by Python as `TesEvent` variants.
5. For every emitted event, strategy `on_event(event)` may emit new `TesCommand` values.
6. Loop continues until no pending commands remain.
7. CLI prints a deterministic summary for the run.

This preserves TES contract boundaries: strategies produce strict command models, engine output is consumed as strict event models, and Python-visible event shape remains normalized at serialization boundaries.

## Current built-in strategy

### `simple_market_maker`

The currently wired strategy name is:

- `simple_market_maker`

`simple_market_maker` currently posts one buy limit order and one sell limit order from `on_start()`, then emits no follow-up commands in `on_event(event)`.

Run it with:

```bash
./tes sim run --strategy simple_market_maker
```

Typical summary output includes strategy name, total emitted events, and total trades.

## Current strategy command

The current strategy-run CLI entry point is:

```bash
./tes sim run --strategy simple_market_maker
```

Today, unknown strategy names are rejected by the loader with a non-zero exit.

## Planned workflow extensions

The following items are planned and not yet the current interface in this repository state:

- `./tes sim list-strategies` for discoverable strategy listing.
- Config-driven strategy runs so strategy selection and parameters can be supplied via structured run configuration instead of only CLI flags.

Until those are added, strategy runs are name-based via `--strategy` and currently include the built-in `simple_market_maker` path above.
